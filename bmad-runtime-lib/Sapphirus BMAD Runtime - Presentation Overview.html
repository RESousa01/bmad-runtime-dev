The choice

Same principles. Different place of work.

The Web App is a managed team environment. Windows Desktop is a local-first coding companion. They share BMAD and Airlock, but they do not mix workspace or execution models.

Web App

Managed in Azure
Best for

Distributed teams, shared projects, central review, and organisation-wide governance.

Getting started

Open the browser, sign in with your work account, and join a managed project. Nothing to install.

What it feels like

A consistent workspace available from any supported browser, with team approvals and activity in one place.

Advantages

  • No installation
  • Easy team sharing and review
  • Central policies and evidence
  • Consistent managed environment

Trade-offs

  • Internet connection required
  • The repository must be connected or uploaded to Azure
  • Remote work may add some waiting time

Windows Desktop

Local-first on Windows
Best for

Developers working with local repositories, sensitive code, existing tools, and fast edit-test loops.

Getting started

Install the signed Windows app, sign in, and choose exactly which workspace folders Sapphirus may use.

What it feels like

A coding agent beside your local tools: inspect files, review diffs, approve edits, run tests, and roll back.

Advantages

  • No full repository upload for normal work
  • Files and ordinary edits stay local
  • Uses your existing Windows toolchain
  • No Docker, server, GPU, or local AI model

Trade-offs

  • Windows installation and updates
  • Selected context still goes to Azure AI
  • Collaboration and sync are opt-in
  • Local commands use your device resources

The selected emphasis changes only this presentation view. In the product, a project keeps one delivery model from start to finish.

Files and development

What happens to the project in each experience?

The biggest difference is not the AI. It is where the working copy lives and where approved edits, commands, and tests happen.

Web App

Cloud development
01
Bring the project into Azure

Connect an approved repository or upload a project archive. The browser does not automatically read folders from the user's computer.

02
Work from a managed cloud copy

Sapphirus creates a controlled workspace that the team can open, review, and return to from supported browsers.

03
AI reads selected cloud context

Relevant files or snippets are selected from that managed workspace and sent to the approved Azure AI service.

04
Edits and tests run remotely

After approval, changes and commands run in an isolated Azure environment rather than on the user's computer.

05
Results are ready for the team

Checkpoints, evidence, approvals, and outputs are centrally available. Export or repository updates remain controlled actions.

Windows Desktop

Local development
01
Choose a local workspace folder

The user selects exactly which project folder Sapphirus may use. Normal work does not begin with a full repository upload.

02
Keep the working copy on the device

The app reads and edits files inside the selected workspace while the repository remains in its existing local location.

03
Send only the context needed by the AI

Selected and redacted snippets are sent to Azure AI for reasoning. The user can see which files contribute to the request.

04
Edits and tests run locally

After approval, the app changes local files and uses the installed Windows toolchain to run commands or tests.

05
Local recovery comes first

Checkpoints, diffs, evidence, and rollback are stored locally. Sync or a remote job happens only when the user chooses it.

Web App

The project is cloned or uploaded into a managed Azure workspace before development work runs.

Windows Desktop

The repository stays local by default; there is no automatic whole-project upload.

Both experiences

The approved Azure AI service receives the relevant context needed to answer, subject to policy and redaction.

The shared experience

From idea to outcome, the trust model stays familiar.

People always see what the AI proposes before important changes happen. The difference is simply where the approved work runs.

BMAD guides

Define the objective, context, and expected result.

AI proposes

Get a clear plan, proposed changes, and expected checks.

Airlock checks

Review the exact diff or command and approve with guardrails.

Work executes

In Azure for Web App, or locally for Windows Desktop.

Evidence remains

See what changed, what passed, and how to roll back.

The AI proposes. The user approves. Sapphirus records the result and the recovery point.

Recommendation

Choose based on the work, not the technology.

Start with the audience that has the clearest need. Both products can exist in the portfolio, but each project should keep one consistent delivery model.

Lead with Web App

  • Teams need access from different devices or locations.
  • Shared review, visibility, and central policy are priorities.
  • The organisation is comfortable managing source in Azure.
  • A no-install experience matters more than the fastest local loop.

Lead with Windows Desktop

  • Developers already work from local Windows repositories.
  • Source should remain local except for selected AI context.
  • Fast local edits, commands, and tests are central to the experience.
  • Collaboration can be optional rather than always on.
Current emphasis: Web App

Best when the first priority is a managed, accessible team workspace with central collaboration and governance.
