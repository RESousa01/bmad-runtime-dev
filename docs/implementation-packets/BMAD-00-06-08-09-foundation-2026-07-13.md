# BMAD Method Foundation, Architect Slice, and Inactive Builder Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use `superpowers:subagent-driven-development` (recommended) or `superpowers:executing-plans` to implement this plan packet-by-packet. Every implementation step uses a checkbox for durable progress tracking.

**Goal:** Make BMAD Method the source-bound conversational foundation of the Windows desktop Agent, prove it with both `bmad-help` and one command-free Architect/Architecture vertical slice, and add BMAD Builder as an essential but strictly inactive authoring surface for stateless agents and simple inline workflows.

**Architecture:** JSON Schema 2020-12 defines the shared wire shapes, while handwritten Rust owns all `windows_local` state, semantic validation, model binding, and lifecycle transitions. Method and Builder semantics are independently implemented as reviewed, repository-owned descriptors and managed instruction projections; no file from the external context library is copied into the product repository or consumed by build, test, runtime, CI, Tauri resources, or distribution. The renderer consumes bounded projections over typed IPC and can neither mint BMAD authority nor activate generated content.

**Tech stack:** Rust `1.97.0` edition 2021; Tauri core `2.11.5`; Node `24.18.0`; pnpm `11.12.0`; TypeScript **exactly `7.0.2`**; React `19.2.7`; JSON Schema 2020-12; generated Rust/TypeScript/C# boundary types; SQLite plus encrypted CAS under the signed Rust host.

## Global constraints

- This is an internal-organization Windows desktop product. “Release” in this packet means promotion to an internal signed deployment ring; it does not mean a public marketplace, public SaaS, or public package publication.
- `deliveryModel` is always `windows_local`; the signed Rust host and local store are the sole local lifecycle authority.
- BMAD Method owns method, workflow, artifact, help, customization, and working-state semantics. BMAD Builder owns authoring and analysis semantics. Sapphirus owns identity, policy, durable state, model routing, egress consent, local effects, evidence, promotion, and activation.
- `bmad-runtime-lib` remains an optional external context library. Final code, tests, generated artifacts, package manifests, Tauri resources, CI, installers, and runtime lookup paths must work when that folder is absent.
- Reviewed upstream scripts are reference material only. No upstream setup, render, eval, scaffold, cleanup, wake, hook, dependency-install, or candidate-provided script may execute in these packets.
- Commands and child-process execution remain absent. No shell, terminal, command schema, command IPC, script runner, or hidden experimental capability is introduced.
- There is no generic workflow AST. Method instructions remain prompt-native and use an explicit `SkillExecutionProfile`; Builder workflows remain inline-first with descriptive progressive-disclosure resources.
- Agent Builder preserves source action `create_rebuild`, `edit`, and `analyze`. Workflow Builder preserves `build`, `edit`, and `analyze`. `Convert` is absent. `Analyze` produces review evidence and never claims evaluation, rehearsal, registration, promotion, or activation.
- The early Builder output set is exactly one stateless-agent shape and one simple inline-workflow shape. Memory, autonomous/Pulse, module, evaluation, rehearsal, publication, promotion, activation, and self-modifying behavior remain unavailable.
- Builder revisions live in app-owned encrypted CAS. They do not write a selected workspace, register a capability, or change `ToolAvailabilitySnapshot`.
- Every model invocation receives a freshly reviewed `ContextEgressManifest` and a single-use context decision. The owning Rust service atomically consumes that decision for exactly one model invocation; a second invocation requires a new review and new decision even when the selected bytes and every manifest field are identical. Any drift in item bytes, hashes, classification, redaction, model/deployment/region, retention, schema/profile hash, or consent hash voids the unconsumed decision.
- Provider credentials never exist on the device. Provider calls go through the organization’s desktop support plane with `store=false`, hosted tools off, background work off, and no whole-repository upload.
- All authority- or security-bearing objects are closed, use explicit discriminators, reject duplicate keys before schema validation, and fail closed on unknown majors, discriminators, profiles, or security enums.
- Contract hashes use RFC 8785 JCS and the exact note-99 preimage: `sha256(UTF8("sapphirus:" + purpose + ":" + schemaMajor + "\n" + JCS(value)))`.
- Writers emit only the negotiated current major. Readers support the current major plus as many as two immediately preceding **released** majors when those majors actually exist, through pure upcasters. The first BMAD family is v1, so it has no fictional M-1/M-2 fixtures; it proves v1 plus v2/unknown-major fail-closed behavior. Each later major adds real prior-major fixtures without rewriting immutable source bytes, and no upcaster executes effects.
- TypeScript application and generated-contract compilation use `7.0.2`. This packet introduces no TypeScript 6 sidecar or compiler-API compatibility package.
- UI vocabulary remains Session, Agent, Task, Local workspace, Proposed changes, Review changes, Apply changes, Revise, Discard, and Undo changes. The product must not display “Chat,” “Approve & apply locally,” “Run,” “Execute,” or “Convert.”
- Odysseus is clean-room UX inspiration only. No Odysseus source code, asset, string table, or theme is copied.
- Every changed line receives an implementer full-diff review and an independent bug-focused review. Every packet in this plan also requires the stated adversarial review before merge.

### Exact early Builder limits

These are contract constants and Rust enforcement limits, not configurable renderer hints. The stricter profile inventory still applies: a simple workflow contains exactly one `SKILL.md`. | Boundary | Exact cap | Failure |
|---|---:|---|
| User create/revision/analyze request | 32,768 UTF-8 bytes | `builder_request_too_large` |
| Serialized model response before duplicate-key parsing | 2,097,152 bytes | `builder_model_response_too_large` |
| JSON nesting depth | 16 | `builder_output_nesting_exceeded` |
| Proposed files | 16 | `builder_output_file_limit_exceeded` |
| One decoded UTF-8 file | 262,144 bytes | `builder_output_file_too_large` |
| Sum of decoded proposed file bytes | 1,048,576 bytes | `builder_output_total_too_large` |
| Normalized relative path | 240 UTF-8 bytes, at most 16 segments | `builder_output_path_too_long` |
| One normalized path segment | 120 UTF-8 bytes | `builder_output_path_segment_too_long` |
| Deterministic plus model-lens findings retained per analysis | 512 | `builder_analysis_finding_limit_exceeded` |

Limits are checked before allocation where possible and again over decoded values. ### Contract invariants

1. `BmadCapabilityKey` is the tuple `packageVersionId + moduleCode + skillName + normalizedAction`; `normalizedAction` is an explicit nullable field and is null only for a single-action skill.
2. `InstalledSkillRecord[]` and `BmadHelpActionRecord[]` are separate collections. A help row cannot fabricate an installed skill, and an installed skill does not automatically become a user-facing action.
3. `BmadAgentRoster` is a third separate collection, canonically sorted by `moduleCode + agentCode`. Each `BmadAgentRecord` binds its module source hash, agent code/name/title/icon/team/description, exact persona/customization graph hash, and source-ordered closed `BmadAgentMenuItem[]`; menu codes are unique within the agent. Each menu item has exactly one explicitly discriminated `BmadAgentMenuTarget`: `skill_target` binds a complete `BmadCapabilityKey` and must resolve to either an `InstalledSkillRecord` or explicit `BmadDependencyAvailability`; `prompt_reference` binds an exact normalized source-local member label and source-member hash from the reviewed customization graph and is always projected `unavailable_source_prompt` in these packets. A prompt-reference target cannot be coerced into a capability key, expose its prompt/body, or dispatch. Roster display metadata grants no capability; a missing, transplanted, ambiguous, or dual-kind target fails closed.
4. A help record preserves the raw 13-column row and a normalized interpretation of `phase`, `precededBy`, `followedBy`, `required`, `outputLocation`, and `outputs`. Menu codes are scoped aliases, never IDs.
5. `BmadInstructionProjection` binds the exact source entrypoint/resource hashes to a reviewed `SapphirusManagedV1` instruction hash. Every source section/resource is classified `adopt`, `adapt`, `defer`, or `reject`; the projection lists blocked tool intents and the typed host input that replaces each adapted operational directive. Raw source text is evidence, not system policy.
6. `SkillExecutionProfile` declares resource timing, invocation modes, required runtimes, tool intents, state hints, completion evidence, customization, and validation profile. It is descriptive and grants no tool or effect authority.
7. The three config graphs remain distinct: Method central four-layer TOML, per-skill customization TOML, and profile-specific compatibility YAML. A schema or resolver cannot flatten them into one precedence list.
8. Method central and per-skill TOML use exact source semantics: scalars replace; tables recursively merge; arrays of tables merge by `code` or `id` only when every combined item has that key; all other arrays append. There is no deletion operator.
9. `BuilderAuthoringAction` is a closed discriminated union:
   - `{ builderKind: "agent", action: "create_rebuild" | "edit" | "analyze" }`
   - `{ builderKind: "workflow", action: "build" | "edit" | "analyze" }`
   `convert`, generic `build` for agents, `evaluate`, `install`, `register`, `promote`, and `activate` are not authoring-action values.
10. The early Builder schema defines only the inactive authoring aggregate and its immutable revision/analysis evidence. Registration, rehearsal, evaluation, publication, promotion, activation, rollback, memory, autonomous, and module object types, state enums, operations, fields, and roots are absent rather than modeled as inert values. No authoring `status` field can be transplanted into a future lifecycle.
11. `BuilderAnalysisRun` records an ordered deterministic-static finding set and, only when separately consented, a closed source-grounded `model_lens` finding set. Agent Analyze uses exactly six stateless lenses; Workflow Analyze uses exactly five. Every lens result binds the analyzed revision, source-member/projection set, deterministic-facts hash, model/deployment/profile/schema/consent hashes, and `evaluationClaim: "none"`. Missing consent/connectivity produces an explicit deterministic-only analysis; neither form constitutes or satisfies evaluation.
12. `BuilderProposedFileSet` carries the exact section-2 limit profile identifier. Structural validation and handwritten Rust enforcement reject any response that exceeds a byte/file/path/depth/finding cap; the renderer/model cannot raise a cap.
13. Deferred capability types and operation names are absent from early renderer-generated TypeScript, standalone validators, IPC schemas/catalogs, and release-binary command registration. Host-owned availability projections may state closed current facts such as `evaluation_unavailable` or `builder_target_deferred`; those facts contain no future lifecycle DTO, transition, authority, or permissive unknown fallback.
14. Every Method session binds exact package, source, managed-instruction projection, skill, action, distribution/install/execution/validation profiles, config/customization hashes, resource set hash, context digest, model/profile/schema hashes, artifact expectations, and either `no_agent` or an exact roster/agent/persona/menu binding. Every checkpoint binds a monotonically increasing turn ordinal, current capability-specific step key, optional next step key, and immutable working-artifact refs. Each accepted context review creates a host-owned single-use decision ID and `BmadContextDecisionConsumption`; the owning Method or Builder service atomically consumes it while binding exactly one invocation ID, manifest hash, consent hash, and all model/schema/profile inputs. Consumed decisions cannot be replayed, including for byte-identical context. A drifted binding voids any unconsumed decision and creates a new review/checkpoint; it is not silently updated.
15. Package text, customization hooks, persistent facts, artifacts, and model output remain untrusted context. They cannot set delivery, authority, owner, policy, egress, approval, spec, tool availability, agent identity, step authority, or activation fields.

## 4. Shared review and verification protocol

Each packet is the smallest independently rejectable behavioral surface. Do not combine two packet IDs in one implementation commit. A packet can land only after its predecessor’s acceptance criteria are recorded.

### Required review records

Each implementation creates these records:

- `docs/reviews/bmad/<packet-id>-implementer.md` — full-diff self-review covering every changed line twice for correctness, error handling, naming, scope, tests, and generated output.
- `docs/reviews/bmad/<packet-id>-independent.md` — a fresh reviewer’s findings-first review for regression, boundary bypass, contract drift, unsafe assumptions, missing tests, and unrelated changes.
- `docs/reviews/bmad/<packet-id>-adversarial.md` — the packet-specific abuse review listed below.

Generated code receives the same two reviews, deterministic regeneration twice, compile/conformance checks, and a generated-tree digest comparison. A reviewer never approves from a summary alone; the full diff and tests are inspected.

### Common stop conditions

Stop the packet and record the blocker if any of these occurs:

- a build, package, test, CI, Tauri resource, or runtime path depends on `bmad-runtime-lib`;
- an upstream or candidate script must run to make the packet work;
- a generic workflow AST or universal numbered-step engine is introduced;
- Builder exposes Convert or any evaluation/install/registration/promotion/activation/memory/autonomy/scheduling/network/command/script operation, DTO, state, or capability; `evaluationClaim: "none"` is the sole early analysis disclaimer and grants nothing;
- a renderer payload can set package authority, session binding, lifecycle state, policy, tool availability, or model/provider endpoint;
- Method or Builder bytes drift after consent/binding without forcing a new review;
- any consumed context decision authorizes a distinct second model invocation, including one with byte-identical context;
- TypeScript resolves to a version other than `7.0.2`;
- any generated language disagrees on canonical bytes, hash, discriminator, nullability, or invalid-fixture reason category;
- a local absolute path, prompt/source content, token, provider credential, encryption key, or raw CAS payload enters renderer persistence, diagnostics, telemetry, or support-plane durable storage;
- an unknown profile, action, schema major, dependency, or lifecycle state is treated as permissive success.

### Common verification commands

Run packet-specific tests first, then the shared gates. The expected result is exit code `0` and no generated or lockfile drift.

```powershell

corepack pnpm --filter @sapphirus/contracts verify:cross-language
```
